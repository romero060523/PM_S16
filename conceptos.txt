CONCEPTOS TÉCNICOS - EventPlanner

¿Qué es MVVM?

MVVM es un patrón de arquitectura que separa el código en tres partes. El Model es donde están los datos, en nuestro caso la clase Event que guarda la información de cada evento. La View es lo que el usuario ve, todas nuestras pantallas de Compose como LoginScreen y HomeScreen. El ViewModel es el intermediario que conecta ambos, contiene la lógica y expone los datos de forma que la UI se actualice sola cuando cambian. En nuestro proyecto tenemos AuthViewModel para la autenticación y EventViewModel para manejar los eventos. La ventaja es que cada parte tiene su responsabilidad clara, es más fácil de probar y mantener.

¿Qué es Firebase Auth?

Firebase Auth es el servicio de Google para manejar usuarios. Básicamente te permite que la gente se registre e inicie sesión sin tener que programar toda esa lógica desde cero. Soporta email y contraseña, pero también puede integrarse con Google, Facebook y otros. Lo bueno es que maneja las sesiones automáticamente y cada usuario que se registra recibe un UID único. En nuestra app lo usamos para que los usuarios puedan crear cuenta e iniciar sesión, y ese UID lo guardamos junto con cada evento para saber quién lo creó. Toda la lógica de autenticación está en el AuthViewModel.

¿Qué es Firestore?

Firestore es la base de datos de Firebase. Es NoSQL, así que no tiene tablas como las bases de datos tradicionales, sino que guarda todo en documentos dentro de colecciones. Lo mejor es que funciona en tiempo real, o sea que si alguien crea un evento, todos los que tienen la app abierta lo ven aparecer al instante sin tener que refrescar. En nuestro proyecto guardamos los eventos en una colección llamada "events", cada evento es un documento con campos como título, fecha, descripción y el userId del dueño. Usamos snapshot listeners para que cuando haya cambios en Firestore, la lista de eventos se actualice sola en la pantalla.

¿Qué es Jetpack Compose?

Compose es la forma moderna de hacer interfaces en Android. Antes tenías que usar XML que era un rollo, ahora escribes todo en Kotlin de forma declarativa. Básicamente describes cómo quieres que se vea la pantalla y Compose se encarga del resto. Las funciones se llaman composables y puedes combinarlas para hacer pantallas complejas. Lo bueno es que si cambian los datos, la UI se actualiza sola sin que tengas que hacer nada. En nuestra app todas las pantallas están hechas con Compose, usamos Material3 para los componentes y manejamos el estado con StateFlow para que todo sea reactivo.

¿Qué es un CRUD?

CRUD son las cuatro operaciones básicas que haces con datos: Create para crear, Read para leer, Update para actualizar y Delete para eliminar. Es lo mínimo que necesitas para que una app funcione con datos persistentes. En nuestro proyecto implementamos todas estas operaciones para los eventos. Create es cuando el usuario crea un nuevo evento con el formulario, Read es cuando mostramos la lista de eventos del usuario que viene de Firestore, Update es cuando edita un evento existente y Delete es cuando lo borra. Todo esto lo manejamos en el EventViewModel con funciones específicas para cada operación, y todo se guarda en la colección "events" de Firestore.
